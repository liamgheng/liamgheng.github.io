刚来新公司不久后在做一个通过 websocket 下发指令的需求，发现如果 server 下发指令后如果 client 执行时间超过 20 几秒则 websocket 会断开，并且难以稳定重现，折腾了两天之后没有结果，Leader 看了下我抓的包，在一分钟之内指出了可能的原因，按照这个思路排查果然找到了原因。

这个例子对我启发很大，其实他并不十分了解 websocket 两端的业务逻辑，但是仅凭借分析抓包结果来定位出了原因。

我觉得这是一个通用的解决问题的方法，在后面遇到的一些问题中也多次使用，确实是分析问题一大利器，下面是我用 tcpdump & wireshark 解决过的部分问题。

## MySQL Has Gone Away

### 问题描述

客户偶尔遇到做某项变更时失败，此时 Django ORM 报错「MySQL Has Gone Away」，过几个小时自动恢复。但是读 MySQL 数据的接口没有问题。

### 问题分析

我们 MySQL 是一主一从，从库只做冷备份。

MySQL 不能写，只能读，分析原因是 Django 往从库建连，从库没有写权限导致的。

读了下 RDS 文档，发现 HA 切换之后，DNS 解析出来的 IP 会发生变化（Pre Master -> Current Master），所以猜测是 HA 切换之后某个地方缓存了 DNS 解析导致继续往 Pre Master 建连。

在 Django 所在容器抓了包，发现发起写 MySQL 操作时 Django 依然往 Pre Master 建连，而非新的 Master，并且抓包中没有发现 DNS 查询报文。

根据抓包结果，猜测是容器或者 Django 缓存了 DNS，后者感觉不太可能，所以着重分析前者，最终发现容器中引入了 nscd 来缓存 DNS 解析，但是当前 Cache 失效时间是 2h，调整到 1min 后解决了问题。

## 容器健康检查失败

### 问题描述

容器健康检查失败，原因是健康检查 timeout。

### 问题分析

重启容器后恢复，过一段时间又不健康了。但容器的 CPU 和内存比较健康。

怀疑是 Django 收到的请求太多，把 Django Worker 撑爆了，无法响应新的连接。

在容器抓了包，发现不断往 sls （日志存储系统）发送查询，看了下 sls sdk 代码，发现在查询超时会最多重试 100W 次。。

所以如果请求中需要查询 sls，而此时 sls 不健康查询失败，导致此请求一直查询 sls 不释放 Worker，从而导致可用 Worker 减少，最终耗光了 Django Worker，从而导致无法响应健康检查的请求。

把默认重试 100W 次改为 5 次后解决了问题。

## 总结

上面的问题乍一看好像和网络没啥关系，所以经常不会想到抓包分析。但从我经验看，在没有做变更的情况下出现的问题，大概率是依赖的其他组件出现了问题，此时通过抓包可以快速帮助我们确定排查的大方向。

其实例子还有不少，尤其是最近在处理的 TCP 半连接堆积导致机器软锁死的问题，更能体现 tcpdump 结合 wireshark 的优势，后面会单独写一篇文章。

上述例子如果之前遇到过坑，那么能在短时间内定位到问题，比如之前了解 nscd，那么可能会先 `ps -ef | grep nscd` 来确认 DNS Cache 的问题，但是这种解决办法依赖经验的积累，且难以复用，而抓包是一种通用的解决办法，熟练之后可以解决相当多的一大类问题，所以需要多积累此类工具的使用。
